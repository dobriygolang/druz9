# Contributing to api

Кратко: это модульный Go-монолит на Kratos. Домены изолированы друг от друга — параллельная работа нескольких контрибьюторов не создаёт мерж-конфликтов.

## Как устроен код

```
apps/api/
├── cmd/api/                # entrypoint, bootstrap (shared, ревью мейнтейнера)
├── api/                    # proto-контракты (shared)
├── pkg/api/                # сгенерированный Go код из proto (shared)
├── internal/
│   ├── api/<domain>/       # HTTP/gRPC-хендлеры домена
│   ├── domain/<domain>/    # бизнес-логика домена
│   ├── data/<domain>/      # postgres-репозитории домена
│   ├── app/                # application-layer для сценариев через несколько доменов
│   ├── model/              # ⚠️ shared-модели (постепенно переезжают в internal/domain/<domain>)
│   ├── middleware/         # auth, request-id и т.п. (shared)
│   ├── server/             # транспорт и регистрация роутов (shared)
│   ├── realtime/           # WebSocket-хабы (shared, скоро разделится)
│   └── clients/            # клиенты к внешним сервисам (shared)
└── scripts/
    └── migrations/         # goose-миграции
```

## Правила, которые проверяет линтер

`make lint` (golangci-lint с depguard) запрещает:

1. **Cross-domain импорты в `internal/domain/`.**
   `internal/domain/arena/*` не может импортировать `internal/domain/profile/*` и наоборот.
   Если нужны данные другого домена — принимайте зависимость через интерфейс в аргументах конструктора сервиса, а не импортируйте чужой пакет.

2. **Cross-domain импорты в `internal/data/`.**
   Репозитории разных доменов не общаются напрямую. Исключение — `internal/data/codetasks` (read-only каталог задач, используется ареной и code_editor).

Если правило мешает решить реальную задачу — не выключайте линтер, обсудите границу с мейнтейнером.

## Куда класть новые типы

- **Тип нужен только одному домену?** → `internal/domain/<name>/model.go` (или отдельный файл рядом).
  Пример: новое поле на `ArenaMatch` — правите `internal/domain/arena/`, а не общий `internal/model/`.
- **Тип реально используется из нескольких доменов или shared-инфры?** → `internal/model/`.
  Обычно это типы вокруг аутентификации (`User`, `Session`), S3 (`UploadObjectRequest`) или кросс-доменной сущности вроде `ProfileProgress`. Если вы добавляете что-то сюда — это архитектурное решение, обсудите с мейнтейнером.
- **Не уверены?** Поставьте в свой домен. Вынести в `internal/model/` потом проще, чем загнать обратно из общего пакета, когда его начинают использовать другие домены.

## Как добавить новый домен

1. Создайте папки: `internal/domain/<name>/`, `internal/data/<name>/`, `internal/api/<name>/`.
2. Добавьте proto-контракт в `api/<name>/v1/<name>.proto`, сгенерируйте стабы: `make generate`.
3. Зарегистрируйте сервисы в `cmd/api/bootstrap_services.go` и хендлеры в `cmd/api/bootstrap_transports.go` через `registerAPIServices`.
4. Добавьте запись в `.github/CODEOWNERS` с владельцем домена.
5. Добавьте deny-правило в `.golangci.yml` для изоляции нового домена.
6. Миграции — через `make migrate-create name=<description>` (timestamp-имена).

## HTTP-ручки только через proto

Все публичные ручки фронта описываются в `api/<domain>/v1/*.proto` с `option (google.api.http)` и генерируются через `make generate`. Реализация — в `internal/api/<domain>/`.

**Не добавляйте** inline handlers в `cmd/api/bootstrap_transports.go` через `r.POST(...)` / `r.GET(...)`. Это приводит к:
- рассинхрону контракта и документации (`docs/openapi.yaml` генерится из proto),
- отсутствию типобезопасности на клиенте (фронт не получает сгенерированных типов),
- ручному auth/JSON decode в каждом месте,
- мерж-конфликтам в одном большом bootstrap-файле.

Единственное оправданное исключение — нестандартные responses (binary image, SSE stream, WebSocket upgrade). Пример в текущем коде: `GET /api/v1/profile/avatar/{user_id}` — отдаёт `image/jpeg`.

## Как работать над существующим доменом

- Ревьюер назначается автоматически из `.github/CODEOWNERS`.
- Меняйте только файлы своего домена. Если задача требует `shared`-кода (`cmd/`, `model/`, `middleware/`) — это ок, но такой PR дополнительно ревьюит мейнтейнер.
- Тесты — рядом с кодом (`*_test.go`). Моки — в `internal/domain/<name>/mocks/`, не делитесь моками между доменами.

## Миграции БД

Миграции хранятся в `scripts/migrations/`, применяются через `goose`.

**Всегда создавайте миграцию через Makefile:**

```bash
make migrate-create name=add_arena_daily_streak
```

Команда генерит файл с timestamp-именем (`20260417131500_add_arena_daily_streak.sql`). Так два контрибьютора, работающих параллельно, не получат одинаковый номер.

**Не создавайте миграции с sequential-номером** (`00028_xxx.sql`, `00029_xxx.sql`) — это гарантированный мерж-конфликт с чьим-то соседним PR. Существующие `00001–00027` — наследство до этого правила, их не трогаем.

Заполняйте секции `-- +goose Up` и `-- +goose Down` — наличие `Down` обязательно, иначе откатить миграцию будет нельзя.

## Перед PR

```bash
make lint              # golangci-lint + архитектурные правила
go test ./...          # юниты
make generate          # если меняли proto
make migrate-up        # если добавляли миграцию (на локальной БД)
```

## Границы, которые ещё не автоматизированы

Ниже — зоны, где мерж-конфликты всё ещё возможны. Работаем над ними поэтапно:

- `internal/model/*.go` — частично shared. Все чисто-доменные типы должны создаваться в своём домене (см. «Куда класть новые типы»).
  Исторические файлы (`arena.go`, `code_editor.go`, `interview_prep.go` и др.) содержат смесь shared- и доменных типов — разлучать связанные сущности без необходимости не стоит, оставляем как есть.
- `scripts/migrations/` — историческая sequential-нумерация (00001–00027). Новые миграции идут в timestamp-формате через `make migrate-create` — конфликтов имен больше не будет. Per-domain разделение с отдельными goose-таблицами — возможная дальнейшая оптимизация, но требует сложной миграции состояния прода, пока не делаем.
- `cmd/api/bootstrap_*.go` — центральная регистрация. Все публичные ручки теперь описываются в proto и регистрируются централизованно через `registerAPIServices` (см. выше).
- `internal/realtime/` — разделён на `realtime/arena/` и `realtime/codeeditor/`; общие схемы сообщений — в `realtime/schema/`. Новые realtime-фичи идут в свой подпакет.

Пока этих задач не сделали: если трогаете эти файлы, постарайтесь минимизировать дифф и договориться с другими контрибьюторами о последовательности мержей.
